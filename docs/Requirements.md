# 요구사항 정의서

## 정책
- 포스팅 수 수집은 월요일 00시 01분에 수행한다. (블로그 포스팅 완료의 기준이 월요일 00시 00분 ~ 일요일 23시59분 이므로)
- 인원마다 블로그 카테고리는 한 가지만 가진다.

## 1. 기능 요구사항
### 1.1 인원 관리
- 인원 추가/수정/삭제/목록 조회

### 1.2 포스팅 수 관리
- 매주 월요일 0시에 배치를 통해 유저별 한 주 포스팅 수 수집
- 유저별 포스팅 수 조회
- 해당 테이블 정렬 및 검색 기능

## 2. 비기능 요구사항
## 2.1 성능
- 단일 요청 응답 시간 2초 이내
- 중복 요청 방지 처리 필요
- 캐시 기반 응답 구조 (Caffeine 사용 계획)

## 2.2 안정성
- 캐시는 인메모리 기반으로 운영된다.
- 애플리케이션 기동 시 캐시 Warm-up 전략을 선택적으로 수행
- 캐시 미스 발생 시 DB 조회 후 캐시에 적재
- 유저 삭제는 soft delete 로 처리

## 2.3 확장성
- AWS 이전 가능한 구조
- Docker 기반 배포

# 3. 예상 장애 포인트 및 대응 전략
## 3.1 로컬 캐시 사용
문제:
- 서버 재기동 시 캐시 비어있음 -> 초기 조회 지연 발생
대응:
- 애플리케이션 기동 시 캐시 Warm-up 전략을 선택적으로 수행

## 3.2 중복 요청 문제
문제:
- 응답 지연 시 동일 요청 반복 발생
방지 전략:
- 프론트엔드에서 요청 중 버튼 비활성화 처리
- 조회 API는 Caffeine 기반 캐시 적용
